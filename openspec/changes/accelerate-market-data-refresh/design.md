## Context

首页资产目录当前由浏览器直接执行 `assetCount → offeringCount → getAsset[] → getOffering[]` 四段链上读取。即使 RPC 单次往返约 1 秒，依赖瀑布也会将热刷新后的首批表格数据推迟到约 4.7 秒；硬刷新时 React Query 内存缓存丢失，页面只能重新显示骨架。另一方面，`useMarketEvents` 需要从部署区块增量扫描事件，事件未就绪时首页用一个联合 `isLoading` 阻塞全部统计指标。

项目已有 `GET /v1/assets` 公共端点、React Query、精确 predicate 失效与分层轮询机制。本变更必须保持现有 API 结构和 RPC 请求预算，不引入需要运维的新外部存储。

## Goals / Non-Goals

**Goals:**

- 将服务端资产快照的链上读取从四段串行缩短为两轮并行读取。
- 在单个服务实例内复用 15 秒快照，并合并同一时刻的并发冷请求。
- 让浏览器硬刷新时可立即恢复最后一次成功资产快照，同时后台校准。
- 让所有标准交易入口在交易确认后立即触发精确失效。
- 让只依赖资产的数据先显示，不再等待事件索引。

**Non-Goals:**

- 本轮不建设跨实例 Redis、数据库或长期事件索引服务。
- 不更改 `/v1/assets` 的 JSON 响应结构和链上数据口径。
- 不缩短现有 HOT/WARM/COLD 直接链上轮询间隔，不增加稳态 RPC 预算。
- 不以模拟数据替代尚未完成的链上事件扫描。

## Decisions

### 1. 使用进程内 TTL 快照与 in-flight promise 合并

`GET /v1/assets` 通过模块级缓存保存最近一次成功 payload 和生成时间，TTL 固定为 15 秒。缓存过期后的首个请求创建加载 promise，后续并发请求复用该 promise；失败不覆盖最后一次成功快照。

选择该方案是因为它不新增基础设施，并能消除同一实例内的重复链上读取。备选的 Redis/KV 能跨实例共享，但需要新的部署依赖与一致性策略，留给后续长期事件索引方案统一评估。

### 2. 将独立链上读取并行化为两轮

第一轮并行读取 `assetCount` 和 `offeringCount`；获得数量后，第二轮并行执行资产 multicall 与发行 multicall。两组结果仍沿用现有容错映射和静态元数据补全，保持响应兼容。

### 3. 首页通过持久化 SWR hook 消费公共快照

新增首页专用 React Query hook，以稳定 query key 请求 `/v1/assets`。最后一次成功结果按版本化 key 写入 `localStorage`，刷新时作为 `initialData` 恢复；query 在后台按 15 秒可见性间隔重新验证。请求失败时保留 last-good 数据，仅在没有任何可用快照时进入错误/骨架状态。

资产详情、投资组合和助手继续使用现有直接链上 hooks，避免改变需要钱包上下文的数据路径。

### 4. 失效职责放在交易 hooks

六个标准交易 hooks 在 `status` 首次进入 `success` 时调用共享失效 hook，按 buy/sell/cancel/claim 复用现有 predicate。资产类 predicate 额外匹配首页快照 query key。助手只消费交易 hook 状态，不再承担缓存一致性职责，避免其他入口遗漏和助手入口重复失效。

### 5. 统计条采用字段级加载状态

`AVG YIELD` 与 `ASSETS` 只依赖资产快照，资产可用后立即显示；`TVL`、`24H VOL` 与 `TRADES` 依赖事件，在事件索引完成前继续显示骨架。这样不伪造事件数据，也不会让慢事件路径阻塞已知资产指标。

## Risks / Trade-offs

- [多实例之间最多 15 秒不一致] → 保持 `s-maxage=15`，交易成功后客户端立即绕过陈旧 UI；该窗口符合公共市场快照的时效目标。
- [localStorage 结构升级产生旧数据] → 使用带 schema 版本的 key，并在恢复时严格校验、解析失败即丢弃。
- [交易状态重复渲染导致重复失效] → 共享 hook 记录已处理的成功交易标识，同一成功状态只执行一次。
- [后台刷新失败留下较旧快照] → UI 保留 last-good 并暴露错误；下一次可见性轮询继续校准，不把失败结果写入缓存。
- [事件指标仍可能较慢] → 本轮只解除其对资产目录和资产指标的阻塞；持久化事件索引作为独立后续架构变更。

## Migration Plan

1. 先加入纯函数与缓存行为测试，再实现服务端快照。
2. 加入客户端解析、恢复和 last-good 测试，再切换首页数据源。
3. 将即时失效迁移到交易 hooks，并删除助手中的重复调用。
4. 分拆统计字段加载状态，运行单元测试、类型检查、生产构建和浏览器性能复测。
5. 如需回滚，首页可恢复使用 `useAssets`；公共端点缓存层可独立移除，API 格式无需迁移。

## Open Questions
