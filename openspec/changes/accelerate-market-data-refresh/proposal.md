## Why

市场首页目前在每次冷启动时串行读取资产数量、发行数量、资产与发行信息，硬刷新后没有可复用的上一份数据；同时交易成功后的缓存失效只存在于助手入口，导致常规交易入口需要等待下一轮轮询。实际测量中，热刷新后资产表仍需约 4.7 秒才出现，事件扫描还会让整条统计栏持续显示骨架。

## What Changes

- 为公共资产目录增加 15 秒服务端快照缓存、并发请求合并与链上读取并行化。
- 首页使用持久化的 stale-while-revalidate 数据源：立即展示最后一次成功快照，并在后台校准最新数据。
- 将交易成功后的精确缓存失效下沉到标准交易 hooks，并覆盖首页资产快照查询。
- 将资产派生指标与事件派生指标分开显示加载状态，避免较慢的事件索引阻塞资产数量和平均收益率。
- 保持现有链上事件索引、RPC 轮询预算和公共 API 响应结构兼容；不引入新的外部数据库或缓存服务。

## Capabilities

### New Capabilities

- `market-data-snapshot`: 定义公共资产目录的服务端短期快照、并发合并和客户端持久化 SWR 行为。

### Modified Capabilities

- `web-data-freshness`: 将交易成功失效职责下沉到交易 hooks，扩展资产失效范围到首页快照，并允许资产指标先于事件指标可用。

## Impact

- 受影响代码：`web/lib/api/publicQuery.ts`、`web/app/v1/assets/route.ts`、首页数据 hooks、交易 hooks、失效规则与市场统计组件。
- 受影响 API：`GET /v1/assets` 的 JSON 结构保持兼容，仅增加服务端短期缓存行为。
- 依赖与部署：不新增第三方依赖，不要求 Redis、数据库或额外部署服务；服务实例之间允许在最多 15 秒窗口内存在快照差异。
